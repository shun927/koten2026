# 【旧仕様・参考】座標送信 メッセージ仕様（送信PC→TouchDesigner / UDP / 相対3D）

> **非推奨**: 新規実装は Box平面モード（`docs/requirements_message_format_box_plane.md`）を使ってください。この仕様は既存資産との互換のために残しています。

この仕様は「PC（単一カメラ）で推定した人差し指先端の相対3D」をTouchDesignerへ安定して送るための要件です。

## 1. 目的 / 非目的

### 目的
- touch側でリアルタイム利用できる「相対3D」座標を安定して受信できること
- 通信遅延が溜まりにくく、ロスト/再接続に強いこと

### 非目的
- 実世界の絶対距離（mm）での高精度な奥行き推定
- 複数カメラ/深度センサを用いた計測

## 2. 用語
- `world`: MediaPipe系が返す3D座標（相対尺度であることが多い）
- `landmark index`:
  - `0`: wrist
  - `5`: index_finger_mcp
  - `8`: index_finger_tip（送信対象）
  - `17`: pinky_mcp

## 3. 推奨する座標の作り方（重要）

### 3.1 原点の固定（相対化）
毎フレーム、以下の相対位置を作る：
- `tip_rel = tip_world(8) - wrist_world(0)`

### 3.2 スケール正規化（端末差・距離変化に強くする）
手サイズの指標を作り、相対座標を正規化する（推奨）：
- `scale = ||index_mcp_world(5) - pinky_mcp_world(17)||`
- `tip_norm = tip_rel / scale`

要件：
- `scale` が閾値未満（例: `1e-6`）のときは `tip_norm` を無効化する（ゼロ割防止）
- PC側の利用は基本 `tip_norm` を優先する（`tip_rel` でも可）

### 3.3 平滑化（任意だが推奨）
送信側/受信側どちらか一方で EMA（指数移動平均）を入れる：
- `p = alpha * p + (1-alpha) * new_p`
要件：
- 低遅延用途は `alpha=0.6〜0.85` 程度から調整
- ロスト時は「最後の値を短時間保持 → 失効」できる設計にする

## 4. メッセージ形式

### 4.1 送信レート
要件：
- 目標 30FPS（最低 15FPS 以上）
- 推論フレームと送信フレームを一致させる（間引く場合は等間隔に）

### 4.2 JSON（実装容易・デバッグ向き）
要件：
- 1行JSON（改行区切り）または1パケット1JSON
- 数値は `float`（Python/JSのnumber）

例（1メッセージ）：
```json
{
  "v": 1,
  "t_ms": 1700000000000,
  "seq": 12345,
  "src": "pc",
  "hand_index": 0,
  "hand": "Left|Right|Unknown",
  "conf": 0.0,
  "tip_img": [0.5, 0.5],
  "tip_rel": [0.0, 0.0, 0.0],
  "tip_norm": [0.0, 0.0, 0.0],
  "scale": 1.0,
  "valid": true
}
```

フィールド要件：
- `v`: スキーマバージョン（整数）
- `t_ms`: 送信元での時刻（UNIX epoch ms）
- `seq`: 連番（ドロップ検出用）
- `src`: 送信元ID（例：`pc`）
- `hand_index`: 同一フレーム内の手の番号（0始まり。未検出の心拍は -1）
- `hand`: 利き手推定（なければ `Unknown`）
- `conf`: 0〜1（推定の確信度。取得できなければ 0 固定でも可）
- `tip_img`: 画像座標（正規化0..1）の指先 `(x,y)`。touch側で左右の穴を位置で割り当てる用途に使う
- `valid`: 検出できた/できない（未検出時は false）
- 未検出時は `valid=false` かつ座標は送っても良いが、touch側は無視できること

## 5. 通信要件

### 5.1 推奨プロトコル
- 推奨：UDP（JSON）

### 5.2 UDPの要件
- PC側は「最新 `seq` を採用」し、古い/遅いパケットは破棄できること
- 送信は同一LAN内を前提（NAT越えは非対象）

## 6. 受信側（PC）要件
（TouchDesigner 側で実装する想定）

- メッセージ欠損に強い（一定時間 `valid=false` 継続でロスト扱い）
- 受信DATと処理/出力を分離（ブロックしない）
- ログ（`seq` 欠損率、遅延、FPS）を取れる
